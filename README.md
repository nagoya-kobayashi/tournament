# Indoor Matchboard

文化祭・校内選手権向けの、リーグ戦／トーナメント進行・日程共有Webアプリです。ビルドやパッケージのインストールは不要で、`index.html` をWebサーバーから配信すれば動作します。

## 主な機能

- オセロ、ボッチャ、UNO、コネクト4を初期登録済み（追加・編集・削除可能）
- 競技ごとに方式、会場、1試合の長さ、個別の開始枠を設定
- クラス×競技のエントリー数を0〜3で設定
- 1チームなら `1B`、複数なら `1B①`・`1B②`・`1B③` と自動命名
- リーグ、トーナメント、リーグ＋決勝トーナメントに対応
- 出場枠へチーム、シード、リーグ順位枠を手動割り当て
- チームの同時出場を避け、移動用の1枠も空ける自動日程作成
- 自動作成後の時刻・会場の手動調整と、競合警告
- クラス別・競技別の試合タイムライン（日ごとの開閉状態を端末に保存）
- 勝ち上がった場合の仮予定は白地の点線枠で最後尾に配置し、表示／非表示を切り替え可能
- 予定は同時刻・同会場をそろえた最大4列のカード表示で、カードから直接結果入力
- 終了試合はグレーの網掛けと大きな赤字の勝者名で表示
- 組み合わせはトーナメント表上のチーム名プルダウンから編集し、設定済みチームは出場枠を入れ替え
- 試合日程はトーナメント表と時刻×会場表を並べ、試合番号をセルへ配置
- エントリー・組み合わせ・日程の変更時は後続データをロックし、強制変更時だけリセット内容を警告
- PDF風の左右対称レイアウトで、勝ち上がり経路を赤線、敗退チーム名をグレーで示すライブトーナメント表（全体／拡大表示）
- 試合番号／チーム名からの結果・得点入力、修正、取消
- 相手未定の後続試合へ先に結果を入力し、前提試合の確定後にチームを自動反映
- Googleスプレッドシート共有時は3秒間隔で更新を確認
- 同時編集はリビジョン番号とApps Scriptのロックで統合

## すぐ試す

Webサーバーでこのフォルダーを公開し、`index.html` を開きます。Pythonがある環境なら、フォルダー内で次を実行できます。

```powershell
py -m http.server 8765
```

その後、`http://localhost:8765/` を開いてください。最初はブラウザーの `localStorage` に保存する「この端末に保存」モードです。

「★トーナメント」原本の組み合わせ・日程は初期入力済みです。内容を変更する場合の基本順は次のとおりです。

1. 「競技設定」で方式・会場・試合枠を確認する
2. 「エントリー」でクラスと出場数を設定する
3. 「組み合わせ」で対戦表を作成し、必要なら出場枠を調整する
4. 「試合日程」で自動割り振りし、手動で調整する
5. 「ライブ」で試合番号またはチーム名を押して結果を入力する

エントリー数を変更すると、その競技の古い組み合わせ・結果・日程はクリアされます。出場枠やリーグ数を変更した場合も、対戦相手との不整合を防ぐため、その競技の結果・日程をクリアします。

## 「★トーナメント」から取り込んだ初期データ

`★トーナメント/トーナメント.xlsx` の現行R8シートと競技別PDFをもとに、次の内容を初期設定しています。

- クラス: 1A～1H、2A～2H、3A～3Iの25クラス
- エントリー: 全4競技で各クラス2チーム、各競技50チーム
- チーム名: 原本どおり①／②表記（3チーム出場時は③まで）
- 方式: 全競技64枠トーナメント、左右各25チーム・各7シード
- 試合番号: 通常のシード戦は番号から除外し、1～48、3位決定戦49、決勝50
- オセロ: 生徒ホール4面、20分枠
- ボッチャ: AL教室・e-room・西展開教室、15分枠
- UNO: 被服室4面、20分枠
- コネクト4: 特別講義室4面、20分枠
- 日程: 原本の「1日目」「2日目」の全50試合を会場まで割り当て済み

原本には年月日の指定がないため、画面では日付を推測せず「1日目 09:00」の形式で表示します。内部では初回起動日の翌日・翌々日を使って時系列を保持しています。実際の日付が決まったら、競技設定の「個別の試合枠」で変更してください。

原本の日程のうち、勝ち上がる可能性まで保守的に計算すると移動1枠を確保できない組み合わせは「要確認」として表示されます。原本の割り振り自体は変更せず取り込んでいます。

## Googleスプレッドシート共有の設定

1. 保存先にするGoogleスプレッドシートを作成します。
2. スプレッドシートの「拡張機能」→「Apps Script」を開きます。
3. [gas/Code.gs](gas/Code.gs) の内容をApps Scriptの `Code.gs` へ貼り付け、保存します。
4. 関数一覧から `initializeMatchboard` を一度実行し、権限を承認します。
5. 「デプロイ」→「新しいデプロイ」→「ウェブアプリ」を選びます。
6. 「次のユーザーとして実行」は自分、「アクセスできるユーザー」は全員（匿名アクセス可）を選びます。外部サイトから認証画面を挟まずに単純リクエストで読み書きするため、この公開設定が必要です。
7. デプロイ後に表示される `/exec` で終わるURLをコピーします。開発テスト用の `/dev` URLは使いません。
8. Matchboard右上の歯車を押し、「GASウェブアプリURL」へ貼り付けて保存します。
9. 接続後、もう一度歯車を開いて「配布用URLをコピー」を押し、そのURLを利用者へ配布します。

### GAS接続先を埋め込んだ配布用URL

配布用URLは、MatchboardのURLへ `gas` パラメーターとしてGASウェブアプリURLを埋め込んだものです。例:

```text
https://example.jp/matchboard/?gas=https%3A%2F%2Fscript.google.com%2Fmacros%2Fs%2F...%2Fexec#live
```

利用者がこのURLを開くと、埋め込まれたGAS URLを接続先として端末に保存し、共有データを自動で読み込みます。URLのエンコードは「配布用URLをコピー」ボタンが行うため、手作業は不要です。「共有を解除」すると、端末の接続設定と現在のURL内の `gas` パラメーターをどちらも削除します。

アクセスキーは、閲覧履歴や転送先へ残らないよう配布用URLには含めません。GAS側で `ACCESS_KEY` を設定している場合は、配布先の各端末で歯車を開き、アクセスキーを別途入力してください。

Apps Scriptウェブアプリは `doGet(e)` と `doPost(e)` でリクエストを受け、`ContentService` でJSONを返します。Googleの仕様上、レスポンスは `script.googleusercontent.com` の一時URLへリダイレクトされますが、ブラウザーの `fetch` は通常これを追跡します。参考: [Apps Script Web Apps](https://developers.google.com/apps-script/guides/web)、[Content Service](https://developers.google.com/apps-script/guides/content)

### CORSプリフライトを避ける通信

フロントエンドは次の通信だけを行います。

- 読み込み: `GET`
- 保存: `POST` + `URLSearchParams`
- POSTのContent-Type: `application/x-www-form-urlencoded;charset=UTF-8`
- カスタムHTTPヘッダーなし

この構成はCORSの単純リクエストの条件を満たし、`OPTIONS`プリフライトを発生させません。参考: [MDN CORSガイド](https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/CORS)

### 任意のアクセスキー

Apps Scriptの「プロジェクトの設定」→「スクリプト プロパティ」に `ACCESS_KEY` を追加すると、同じキーを入力したMatchboardだけがAPIを利用できます。キーはリクエスト内に含まれるため、これは誤操作を減らすための簡易保護です。機密情報向けの強い認証ではありません。公開URLとアクセスキーは運営者だけで管理してください。学校の管理ポリシーで匿名ウェブアプリを禁止している場合、この構成のままではGASとのクロスオリジン共有は利用できません。

別のスプレッドシートに紐づかないスタンドアロンApps Scriptで使う場合は、スクリプト プロパティ `SPREADSHEET_ID` に保存先スプレッドシートIDを設定します。

## 保存形式

共有データはスプレッドシート内の非表示シート `_MATCHBOARD_DATA` にJSONとして保存します。セルの文字数上限に収まり続けるよう、40,000文字ごとの複数行へ分割しています。通常のシートを大会データで埋めないため、運営用の表と同じスプレッドシートに置いても干渉しません。

各保存にはリビジョン番号が付きます。同時保存を検出した場合、ブラウザーは最新データに未送信の操作を適用し直して再送します。GAS側では `LockService` も使用しています。

## 自動日程の考え方

自動割り振りは、確定チームだけでなく、トーナメントでその試合へ進出する可能性がある全チームを候補として扱います。同じチーム候補を含む試合同士は、同時刻を避けるだけでなく、直後の1枠も空けます。リーグ＋トーナメントでは、順位枠に入る可能性があるリーグ内の全チームを保守的に判定します。

枠不足で全試合を置けない場合、置けなかった試合は未設定のまま残ります。競合する手動変更も禁止はせず、「要確認」として表示します。

## テスト

Webサーバー起動中に次をブラウザーで開きます。

- `tests/initial-data-tests.html`: R8のクラス、全チーム、対戦枠、試合番号1～50、全日程（23項目）
- `tests/model-tests.html`: チーム命名、勝敗伝播、先行結果入力、取消、自動日程、リーグ順位（12項目）
- `tests/ui-smoke-tests.html`: 主要画面、原本PDF準拠の対戦表、赤い勝ち上がり経路、シード・敗退・取消表示、結果モーダル、対戦表上の組み合わせ編集、表形式の日程調整、クラス別・競技別予定、データロック、GAS配布用URL（40項目）

どちらもページ上部が `ALL PASSED` になれば成功です。

## ファイル構成

- `index.html` — アプリ本体
- `styles.css` — レスポンシブ表示とトーナメント描画
- `js/initial-data.js` — 「★トーナメント」原本から取り込んだR8初期データ
- `js/model.js` — エントリー、対戦生成、勝敗伝播、順位、自動日程
- `js/store.js` — ローカル保存、GAS共有、同時編集統合、リアルタイム更新
- `js/app.js` — 各画面と操作
- `gas/Code.gs` — Googleスプレッドシート保存API
- `tests/` — モデルテスト、画面スモークテスト、表示確認用データ
